实现nachos中Kthread类的的join()方法

实验题目
  1. Note that another thread does not have to call join(),but if it is called, it must be called only once.
  2. A thread must finish executing normally whether or not it is joined.
实验要求
  1. KThread.join 函数如果被调用也只能被调用一次，第二次调用 的结果是未知的。
  2. 每个线程都维护自己的资源，（在Linux和Windows操作系统中，线程维护的资源存在于 TCB 中）换言之，每个线程拥有自己的 waitJoinQueue，里面存放了由于自己的 join 而被阻塞无法执行的线程。
    由于我们对每个线程添加了 waitJoinQueue 切里面存有在等 待的线程，故而我们需要修改 finish 函数，在线程完成运行即将终止 时检查其 waitJoinQueue，唤醒队列中在等待的线程。
自己理解
  • join()方法的作用是让另一个线程运行，直到另一个线程运行完毕之后再继续运行自己
  • join()方法的原理:
    首先一个线程A调用join()方法是为了让另一个线程B执行，所以该线程首先会查看自身是否处于运行结束的状态，若线程A处于结束状态则join()方法没有意义直接退出
    若A不处于结束状态，且A是第一次调用join()，则将线程A添加到B的joinqueue队列中，在B执行完,调用finish()方法时，检查自己的joinqueue队列，并将A添加到就绪队列中
  • 疑惑:当前正在运行的线程被中断在哪里被重新加入了就绪队列？？
    解答:参见yield()方法

利用互斥锁直接实现条件变量

实验题目
  1. Provide atomicity by using interrupt enable and disable.
  2. Provide an equivalent implementation without directly using semaphores (you may of course still use locks, even though they indirectly use semaphores).
  3. Your second implementation of condition variables must reside in class nachos.threads.Condition2.
实验要求

许多线程在运行的过程中可能会遇到某个条件的约束，这时它必 须通过这个条件才能继续运行下去。条件变量不能让它们一窝蜂全上，所以设置了一个互斥锁让它们能有一定的次序，一个一个地来。通过了条件变量”考验”的线程可以顺利地执行下一步的操作。

题目要求直接实现，条件变量，不能直接利用信号量。

条件变量：一个条件变量对象代表了线程运行中所需要的一个条件。

  1. 互斥锁必须要有，它是线程们有秩序的接受条件变量检验的保证。

  2. 等待队列必须要有，不满足条件的线程都要挂在上面。

条件变量是一个附属于某个特定条件的变量，利用它线程可以进 行一系列操作。在操作系统运行的过程中，可能会有多个线程会需要 同时使用同一个条件变量。为了避免混乱，对每个条件变量设立一个锁(lock)，只有拿到lock 的线程才可以执行条件变量提供的一系列操作。这样做保证了互斥性，避免了多个线程同时抢夺一个条件变量的 情况。线程在拿到lock 后，可以进行以下操作：

  1. 在线程运行过程中，如果不满足这个特定条件，可以利用对应的条件变量将它阻塞在上面

  2. 在线程运行过程中，如果这个线程产生的某个结果能够提供 这个条件时，可将已经阻塞在对应的条件变量上面的一个线程唤醒。

  3. 扩展(2)的功能，使得阻塞在对应条件变量上面的所有线程 都被唤醒。

为实现上述功能，首先需要在 Condition2 类中创建一个等待队 列，用来存放挂起在条件变量上的线程。

对于 sleep()方法，关中断后将调用该方法的线程挂起到等待队 列上，释放锁后阻塞该线程，最后再申请锁即可。

对于 wake()方法，关中断后判断是否还有线程挂起在等待队列 上，如果有的话将队列头唤醒即可。

对于 wakeAll()方法，它完全建立在 wake()方法的基础上。关中 断后，利用循环判断是否还有线程挂在等待队列上，直至所有阻塞的 线程都被唤醒为止。

自己理解
  • 一个锁，一个等待队列 在执行lock.acquire();的时候acquire()会判断该锁是否被占有
    若被占有，加入lock的等待队列，若没有被占有，检查等待队列为空，设定占用锁的线程为当前线程即可。
  • 若被加入了等待队列，只能等待释放锁的线程唤醒。
  • 若线程执行了condition.sleep()当前线程释放锁，并进入等待队列，在重新获得执行机会时，首先申请获得锁lock.acquire();
  • 当线程执行wake()方法时将等待队列的头线程放入就绪队列。同理wakeall()

实现Alarm类的waitUntil(long x)方法

实验题目
  1. Implemente the waitUntil(long x) method.
  2. There is no requirement that threads start running immediately after waking up; just put them on the ready queue in the timer interrupt handler after they have waited for at least the right amount of time.
  3. Do not fork any additional threads to implement waitUntil().
  4. you need only modify waitUntil() and the timer interrupt handler. waitUntil is not limited to one thread.
实验要求
  1. 新建一个队列
  2. 一个线程调用 waitUntill()时，该线程设定唤醒时间并阻塞， 放入队列中。
  3. Timer 每过 500 个 clock ticks 会产生一个 timeInterrupt。
  4. timeInterupt 时，会运行 handle()线程，将 handler 线程设 置为 将队列中此时应当唤醒的进程加入就绪队列中 即可。
  5. 定义了一个 ThreadInf 类来存储多个 KThread 及其唤醒时间， 属性分别为该 KThread 与其唤醒时刻 waketime。
  6. 定义一个队列 PriorityQueue waitingqueue，存放ThreadInf 。 ThreadInf中实现了CompareTo接口,用 waketime 决定先后顺序，小的在前。
自己理解
  • 一个线程想自己睡一会儿，告诉alarm自己要睡多久，然后alarm算出来应该唤醒该线程的时间，将其加入一个按唤醒时间从小到大排列的队列中，然后线程sleep()
  • alarm有一个总是被时钟间隔一段就唤醒的方法timerInterrupt(),该方法，会检查申请睡觉的线程队列，若空，return(),否则从最早应该叫醒的线程开始唤醒(加入就绪队列)，直到检查到不应该被唤醒的线程，退出

使用条件变量实现同步发送接收消息

实验题目
  1. Implement the Communicator class with operations, void speak(int word) and int listen().
  2. Neither thread may return from listen() or speak() until the word transfer has been made.
  3. Your solution should work even if there are multiple speakers and listeners for the same Communicator
  4. Each communicator should only use exactly one lock.
实验要求

使用条件变量(不能使用 semaphores!)实现一个字长信息的同步发送和接收。使用 void speak(int word)和 int listen()函数来实现 Communicator 类的通讯工作。Speak()函数具有原子性，在相同的 Communicator 类中等待 listen()函数被调用，然后将此字传输给 listen()函数。一旦传送完毕，两个函数都返回(listen 函数返回此字)。

自己理解
  • 首先这是一个生产者和消费者的问题，但是与我们书上说的又不是完全一致，这里还要求了同步，所以生产者还需要等待有人在消费再生产。
  • 生产者角度，生产所诉要的条件，为了保证数据区的安全，生产者首先需要获得所有生产者消费者共用的锁，获得锁之后，先将生产的数量++，然后检查数据区的数据是否有价值，而且此时还有消费者等待消费(不满足条件则sleep(),根据T3的Condition2代码可知，sleep()即释放了锁)。若条件允许后(此时仍然获得锁权限)，写入是数据，声明数据有效，唤醒所有的消费者，生产数量–，完成工作，释放锁
  • 消费者角度，消费者与生产者角度类似，首先获得锁，消费者数量++，检查数据是否有效，无效则唤醒所有的生产者，sleep()，有效，则读出数据，生命数据无效，消费者–，释放锁
  • 本任务基于上一个任务，所以需要搞清楚，sleep()与wakeall()都干了啥

完成PriorityScheduler类实现优先级调度

实验题目
  1. Change a line in nachos.conf that specifies the scheduler class to use.
  2. You must implement the methods getPriority(), getEffectivePriority(), and setPriority().
  3. In choosing which thread to dequeue, the scheduler should always choose a thread of the highest effective priority. If multipl e threads with the same highest priority are waiting, the scheduler should choose the one that has been waiting in the queue the longest.
  4. A partial fix for this problem is to have the waiting thread do nate its priority to the low priority thread while it is holding the lock.
  5. Be sure to implement Scheduler.getEffectivePriority(), which returns the priority of a thread after taking into account all the donations it is receiving.
  6. The Lock class does not need to be modified.
实验要求

Nachos系统已经提供了一个简单的轮转调度器，采用简单的FIFO队列进行调度。优先级调度的传统如下: 每个线程拥有一个优先级(在Nachos中, 优先级是一个0到7之间的整数, 默认为1。在线程调度时, 调度程序选择一个拥有最高优先级的处于就绪状态的线程运行。算法首先要在线程对象中完成对线程优先级修改的函数的实现，方便后面设置修改线程优先级完成试验的过程。

自己理解
  • 首先了解一下先进先出队列的工作流程:
    RoundRobinScheduler继承了Schedule但貌似也没干啥，好吧，谁让他是先进先出，就是一个简单的队列模型，从头取，从尾加，继承的是ThreadQueue
  • 一开始没有好好看英文题目的要求，上去就想到了带排序功能的容器去为就绪态的线程排序，结果人家有要求：优先级一样的，等待时间长的先执行
